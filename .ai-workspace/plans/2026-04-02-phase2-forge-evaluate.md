# Forge Harness — Phase 2: `forge_evaluate` Implementation

## Context

Phase 1 shipped `forge_plan` (v0.3.0) — a planning engine that produces JSON execution plans (stories + ACs with binary shell commands). Phase 2 replaces the `forge_evaluate` placeholder with a real implementation: a **stateless binary grading tool** that runs AC shell commands and produces structured eval reports.

This is the second dogfood of the phased design. The stories and ACs below were generated by the forge_plan pipeline (thorough tier, 2 rounds of critic-corrector). The architecture and file specs are added here for human oversight.

**Key constraint:** forge_evaluate is stateless. It receives a plan + storyId, runs the commands, and returns results. No side effects, no file writes, no persistent state. This makes it safe to run repeatedly (idempotent grading).

## ELI5

We taught the robot to write recipes (Phase 1). Now we're teaching it to taste-test the food — run each check in the recipe and report "this passed" or "this failed" with proof (what the command printed). The taste-tester doesn't cook anything; it only checks.

## Architecture

### Pipeline Overview

```
forge_evaluate(storyId, planPath|planJson, timeoutMs?)
    |
[1] Load execution plan (from file path or inline JSON)
    |
[2] Find story by storyId (throw if not found)
    |
[3] For each AC in story.acceptanceCriteria:
    |   ├── Run AC.command via child_process.exec
    |   ├── Capture exit code, stdout, stderr
    |   ├── Apply timeout (default 30s, configurable)
    |   ├── Use { shell: 'bash' } on Windows only (see D8)
    |   └── Map to CriterionResult { id, status, evidence }
    |
[4] Compute verdict:
    |   ├── All PASS → verdict: "PASS"
    |   ├── Any FAIL → verdict: "FAIL"
    |   ├── Any INCONCLUSIVE (exec error) + no FAIL → verdict: "INCONCLUSIVE"
    |   ├── Zero ACs → verdict: "PASS" (vacuous truth, with warning)
    |
[5] Validate eval report against schema
    |
Return: EvalReport JSON
```

### Key Design Decisions

1. **`child_process.exec`** for command execution — not `spawn`. Commands are shell strings (e.g., `npx tsc --noEmit && echo PASS`), not argv arrays. `exec` runs them through a shell, which is what the planner's ACs expect.

2. **Default timeout: 30 seconds** per AC command. Configurable via `timeoutMs` input parameter (must be a positive number). Long-running commands (e.g., full test suites) may need higher values. Timeout produces status `FAIL` with evidence explaining the timeout. Rationale: in CI contexts, exceeding the time budget is itself a failure condition — the caller's intent is "this command must complete within N ms," and exceeding that is a definitive outcome, not an unknown one. If the caller wants to retry with a longer timeout, the evidence message makes it clear what happened.

3. **INCONCLUSIVE verdict** for execution errors (command binary not found, permission denied, etc.) — distinct from FAIL (command ran but exited non-zero). This matters for debugging: FAIL means "the check didn't pass", INCONCLUSIVE means "we couldn't run the check."

   **Discrimination logic:** In the `child_process.exec` callback, when `error` is non-null: if `error.killed` is `true` (timeout), status is `FAIL`. If `error.code` is a string (e.g., `'ENOENT'`, `'ERR_CHILD_PROCESS_STDIO_MAXBUFFER'`), status is `INCONCLUSIVE`. If `error.code` is a number (non-zero exit code), status is `FAIL`. **Fallback:** if `error` is non-null but none of the above conditions match (e.g., `error.code` is `null` and `error.killed` is `false`, such as a process killed by an external signal where `error.signal` is set), status is `INCONCLUSIVE` with evidence including the full error message. When `error` is null, status is `PASS`.

4. **Evidence format**: Concatenate as `[stdout, stderr].filter(Boolean).join('\n')`. This avoids spurious newlines when one stream is empty and produces `""` when both are empty (consistent with D11). If total length exceeds 4,000 characters, keep the last 4,000 characters (tail truncation) and prepend `'[truncated] '`. Truncation constant `EVIDENCE_CHAR_CAP = 4000`. `NEW_CLAIM: 4000 was chosen as a round number that fits comfortably in a single MCP response while capturing enough context for debugging. Adjust if real-world usage shows it's too short or too long. — source: inference from MCP response size constraints`

   **PASS evidence:** For PASS results, evidence still includes both stdout and stderr (concatenated and truncated per the rule above). Stderr output on a passing command is informational and preserved for debugging.

5. **Sequential execution** of ACs within a story. No parallel execution in Phase 2 — simpler, avoids resource contention. Phase 3 (forge_generate) may parallelize if needed.

6. **No Claude API calls** — forge_evaluate is purely mechanical. It runs shell commands and reports results. No LLM reasoning involved. This is intentional: the "evaluate" primitive is deterministic, unlike "plan" or "generate."

7. **planPath vs planJson** — the tool accepts either a file path to an execution plan JSON, or inline JSON string. `planPath` is preferred for large plans. If both provided, `planJson` takes precedence (explicit over implicit); `planPath` is ignored entirely, even if `planJson` is invalid. If neither provided, error.

8. **Shell selection (platform-conditional)**: On Windows, `child_process.exec` defaults to `cmd.exe`, but AC commands use Unix-style shell syntax (`&&` chains, `grep -q`, `echo PASS`). On Windows, the executor must pass `{ shell: 'bash' }` in exec options to ensure commands run through Git Bash. On Linux/macOS, the executor omits the `shell` option, allowing `exec` to use the system default (`/bin/sh`), which already handles Unix shell syntax correctly. This avoids a hard dependency on bash in minimal containers that only have `sh`.

9. **`maxBuffer` for exec**: `child_process.exec` has a default `maxBuffer` of ~1 MB. Verbose test suites can exceed this, causing exec to kill the process with a buffer overflow error. The executor should set `maxBuffer: 10 * 1024 * 1024` (10 MB) to avoid this. `NEW_CLAIM: 10 MB maxBuffer — source: industry convention for CI/test output capture`

10. **CWD for command execution**: AC commands run with the project root as their working directory. Commands like `npx tsc --noEmit` and `grep -q ... server/types/eval-report.ts` are CWD-sensitive. The executor must accept an optional `cwd` parameter (defaulting to `process.cwd()`), and the MCP tool layer should resolve `planPath` relative to the same CWD. Note: `cwd` is an internal-only option on the executor, not exposed in the MCP tool input schema. MCP callers are assumed to be running within the project directory (the MCP server's own `process.cwd()`). If future use cases require caller-specified CWD, it can be added to the input schema then.

11. **Empty stdout/stderr**: When a command exits 0 with no output, evidence is the empty string `""`. This is a valid PASS — no special handling needed. (Consistent with D4's conditional concatenation: `[stdout, stderr].filter(Boolean).join('\n')` produces `""` when both streams are empty.)

12. **Zero ACs in a story**: If a story has zero acceptance criteria, the verdict is `PASS` (vacuous truth — no criteria to fail). The evaluator should include a warning in the report when this occurs (e.g., a `warnings` array field with `"Story {storyId} has 0 acceptance criteria — PASS verdict is vacuous"`). `NEW_CLAIM: vacuous truth for zero ACs — source: own analysis, consistent with "all PASS → PASS" rule applied to empty set`

## Output Schema

Matches existing `schema/eval-report.schema.json` (with the addition of `INCONCLUSIVE` to criterion status):

```typescript
interface EvalReport {
  storyId: string;
  verdict: "PASS" | "FAIL" | "INCONCLUSIVE";
  criteria: CriterionResult[];
  warnings?: string[];
}

interface CriterionResult {
  id: string;
  status: "PASS" | "FAIL" | "SKIPPED" | "INCONCLUSIVE";
  evidence: string;
}
```

**Schema alignment note:** The existing `schema/eval-report.schema.json` marks `evidence` as optional (not in the criterion `required` array — only `id` and `status` are required). The TypeScript interface above shows `evidence: string` as required. **Resolution:** Update the JSON schema to add `"evidence"` to the criterion `required` array, making it mandatory. Every criterion result should have evidence (even if it's an empty string for silent commands). Also add an optional `warnings` array of strings to the schema.

**Status mapping:**
- Exit code 0 → `PASS`
- Exit code non-zero → `FAIL`
- Timeout (`error.killed === true`) → `FAIL` (evidence: "Command timed out after {N}ms")
- Exec error (`error.code` is a string, e.g., ENOENT) → `INCONCLUSIVE`
- Signal-killed without timeout (`error.code` is null, `error.killed` is false, `error.signal` is set) → `INCONCLUSIVE`
- Fallback (error non-null, no condition matched) → `INCONCLUSIVE`
- Skipped (not implemented in Phase 2, reserved for dependency-based skip) → `SKIPPED`

## Error Handling

| Scenario | Behavior |
|----------|----------|
| Neither planPath nor planJson provided | Return MCP error: "Either planPath or planJson is required" |
| planPath file not found | Return MCP error: "Plan file not found: {path}" |
| planPath is a directory, not a file | Return MCP error: "Plan file not found: {path}" (readFileSync on a directory throws EISDIR — catch and normalize) |
| planPath/planJson is invalid JSON | Return MCP error: "Invalid plan JSON: {parse error}" |
| Both planPath and planJson provided, planJson is invalid JSON | Return MCP error: "Invalid plan JSON: {parse error}" (planJson takes precedence; planPath is ignored) |
| Plan fails schema validation | Return MCP error: "Invalid execution plan: {errors}" |
| storyId not found in plan | Return MCP error: "Story '{storyId}' not found in plan" |
| timeoutMs is zero or negative | Rejected by Zod schema (`z.number().positive()`) before handler is called |
| AC command times out | CriterionResult status: FAIL, evidence: "Command timed out after {N}ms" |
| AC command exec error (ENOENT) | CriterionResult status: INCONCLUSIVE, evidence: "Command execution failed: {error}" |
| AC command killed by signal (not timeout) | CriterionResult status: INCONCLUSIVE, evidence: "Command execution failed: {error}" |
| AC command exits non-zero | CriterionResult status: FAIL, evidence: stdout+stderr (truncated to 4000 chars) |
| AC command exits 0 with no output | CriterionResult status: PASS, evidence: "" |

## Testing Notes

**vi.mock inline factory pattern (P33):** This project uses ESM (`"type": "module"`) with Vitest. When mocking modules like `child_process`, use the inline factory pattern to avoid temporal dead zone errors:

```typescript
vi.mock("node:child_process", () => ({
  exec: vi.fn(),
}));
```

Do NOT use the `vi.mock("node:child_process")` auto-mock form — it does not work reliably with ESM in this project.

**Build requirement for dist/ imports:** Some AC commands (US-01 AC-02, US-02 AC-01/02/03) import from `./dist/...` and require `npm run build` before execution. Run `npm run build` before executing these ACs. Alternatively, consider rewriting these ACs to use vitest tests for consistency (see US-03/US-04 ACs which use vitest and run directly against TypeScript source).

**Unit vs integration tests:** Unit tests in `executor.test.ts` mock `child_process.exec` for deterministic behavior (no real shell calls). The AC commands for US-02 are integration-level acceptance tests that run real commands to verify end-to-end behavior. Both are needed: unit tests for fast, deterministic coverage; ACs for real-world validation.
