# Plan: Build Three-Tier Document System Into forge_plan

## Context

We used `forge_plan` to dogfood the forge-harness implementation. The divergence analysis (`.ai-workspace/plans/2026-04-05-forge-improvement-research.md`, line 9) found 93 items (28 forward + 65 reverse, ~35% divergence). Root cause: `forge_plan` produces monolithic plans that try to be both vision and spec, failing at both.

**This is not a one-time doc restructuring.** The three-tier document system is a product feature to build INTO the forge primitives. Once built, we continue dogfooding.

The existing `docs/forge-harness-plan.md` is NOT being retired — it contains implementation details to be built.

**Persistent backlog:** `docs/primitive-backlog.md` — complete list of ALL items per primitive. Plans capture current scope; the backlog captures everything. Check it when rewriting plans.

Previous research: `.ai-workspace/plans/2026-04-05-forge-improvement-research.md`.

## ELI5

forge_plan writes one giant document that's both the dream and the blueprint, causing ~35% divergence. We're teaching forge_plan to produce three coherent documents from a single human-approved source: the dream (PRD/vision), the chapter list (master plan), and one chapter at a time (phase plans). The human approves the PRD once — everything else flows automatically with built-in evals to catch divergence and self-healing to fix it.

## Design Decisions

### D1: PRD Skill = Vision Doc (Tier 1)

The `/prd` skill already produces exactly what a Vision Doc needs: problem statement, objective, requirements with binary ACs, user workflow, success criteria, out-of-scope — with NO implementation details (enforced by quality checklist and eval checks).

VERIFIED: `/prd` skill found at `~/.claude/skills/prd/SKILL.md`. Quality checklist at `references/quality-checklist.md` flags implementation details. Eval `evals.json` includes check: "No implementation details (HOW) in the PRD body".

**Graceful fallback:** If the `/prd` skill is not installed, `forge_plan` accepts any plain-text vision doc via the `visionDoc` parameter. The `/prd` skill is recommended but not required.

### D2: Human Approves the PRD Only

The human goes through the `/prd` interview, approves the output — that's their one approval. Master plan and phase plans are generated by forge_plan from the PRD automatically. Built-in evals catch divergence between tiers.

**Flow:**
```
Human → /prd interview → approves PRD (ONE approval) →
forge_plan(master) → master plan →
forge_plan(phase, PH-01) → phase 1 plan →
forge_evaluate(coherence) → validate tiers aligned →
implement → forge_evaluate(divergence) → detect gaps →
forge_plan(update) → self-heal → next phase
```

**Failure mode — coherence eval detects gaps:** Re-run `forge_plan` for the failing tier with the coherence findings injected as `implementationNotes`. If the same gap persists after one retry, surface it to the human with the specific finding.

**Failure mode — master plan is wrong:** The human can re-run `forge_plan(master)` with corrective `implementationNotes` at any time. This is not automatic — the human decides when to intervene. The design trades a small risk of master plan errors (caught by coherence eval or during implementation) for eliminating the friction of multi-tier approvals.

### D3: ACs Are Purely Functional — Method-Agnostic

ACs verify WHAT the system does, not HOW. This prevents divergence when the implementer chooses a different approach than the planner assumed.

- **Good:** `curl localhost:3000/api/users | jq '.users | length' | grep -q '^[1-9]'`
- **Bad:** `grep -r "Redis" src/`

**New planner rule:** "ACs verify observable behavior. ACs must NEVER verify implementation method."
**New critic dimension:** "Implementation coupling — flag ACs that grep source for patterns."

**Enforcement tier reality:** The planner rule is Tier 4 (behavioral prose in a prompt) — per P13, expect <50% compliance from the prompt alone. The real enforcement is a post-generation TypeScript regex scan (Tier 1 mechanical check): after the planner emits its JSON, a validator scans each AC `command` field for source-code inspection patterns (`grep -r.*src/`, `rg.*src/`, `find.*src/.*-name`). Any match is flagged as an implementation-coupled AC. This is ~10 lines of code and runs deterministically — no LLM judgment involved. The critic dimension serves as a secondary Tier 4 backup for patterns the regex does not catch.

### D4: Built-In Divergence Detection + Self-Healing

New evaluation modes in forge_evaluate:

**Coherence eval** (between tiers): Does master plan cover all PRD requirements? Do phase plans deliver what master plan promises? LLM-judged.

**Divergence eval** (code vs plan):
- Forward: AC failures (mechanical — existing behavior)
- Reverse: unplanned capabilities in codebase (LLM-judged scan)

**Self-healing rules:**
- **Implementation method divergence** (different approach, same result): update plan to reflect reality. The plan adapts to the implementation. Each method-divergence update is logged in the run record with the original method and the replacement method. If method-divergence updates exceed 50% of stories in a single phase, flag to the human — this signals the plan's architecture assumptions may be fundamentally wrong rather than incidentally different.
- **Functional divergence** (different behavior than specified):
  - Missing functionality: flag as gap, recommend implementation
  - Extra functionality: assess if it aligns with PRD. If yes, update plan to include it. If no, flag as scope creep for human decision.
  - Changed functionality: flag for human review — the PRD is the source of truth for functional requirements.

**Self-healing iteration cap:** Maximum 2 self-healing cycles per phase. If divergence persists after 2 cycles, surface to the human with the full finding history rather than looping indefinitely. `NEW_CLAIM: max 2 self-healing cycles — inference from F9/F18 self-scoring bias risk; unbounded loops waste tokens and may converge on a wrong fix`

**Blocking mechanism for human-review flags:** When running in an automated pipeline, "flag for human review" items are written to `.forge/pending-review.json` and the pipeline pauses that phase. Phases with no dependency on the flagged phase continue independently. A stderr banner is emitted: `BLOCKED: Phase PH-XX has N unresolved review flags — run forge_evaluate --resolve to continue`. If flags remain unresolved, reminders are written to the mailbox with exponential backoff: 24h, 72h, 7d — then stop (3 reminders total over 11 days). This balances notification noise against the risk of a permanently stuck phase. `NEW_CLAIM: .forge/pending-review.json blocking mechanism — own analysis of the automated pipeline gap`

**LLM-eval bias mitigation:** The coherence and divergence evals use structured output (JSON with specific field checks) rather than open-ended prose, reducing the surface area for F9/F18 self-scoring bias. The eval prompts enumerate concrete check dimensions (requirement coverage, AC traceability) rather than asking "is this plan good?". This does not eliminate bias but constrains it to specific dimensions that can be calibrated over time via run records (D7).

**Coherence eval calibration protocol:** Run records track when a coherence eval passes a phase but the subsequent divergence eval finds functional gaps. The metric is: % of coherence-pass phases that later show functional divergence. If this exceeds 30% over a rolling window of 10+ phases, the response is: (1) review the coherence eval prompt dimensions against the missed gaps, (2) add a new check dimension targeting the gap pattern, (3) log the calibration change in the run record. This is a manual review triggered by the metric, not an automatic retune.

**Phase 1 (context injection — low complexity):**

Rationale: the planner system prompt + output schema + codebase summary can collectively reach 15-20k tokens; a 50k-char context budget (~12.5k tokens) keeps the total well under half the working context, preserving signal-to-noise for the planner's reasoning. Revisit after run records show actual token distributions.

### D6: Cross-Cutting Observability for All Primitives

Every forge primitive gets cost estimation, visible progress, and audit trail.

**Cost Estimation (`server/lib/cost.ts`):**
- CostTracker: tokens per stage, pricing multiplier, running total
- Hardcoded pricing + `PRICING_LAST_UPDATED` date
- Advisory budget: `isOverBudget()`, `remainingBudgetUsd()` — does NOT force-stop. Rationale: force-stopping mid-generation produces corrupt partial output (incomplete JSON) that fails validation and wastes all tokens spent so far. Advisory mode lets the caller decide whether to continue or abort cleanly between stages.
- OAuth/Max: "equivalent API cost" label when OAuth detected
- Output: `{inputTokens, outputTokens, estimatedCostUsd, breakdown, isOAuthAuth}`
- **Stale pricing warning:** If `PRICING_LAST_UPDATED` is more than 90 days old, emit a stderr warning: "Pricing data is N days old; estimates may be inaccurate." `NEW_CLAIM: 90-day staleness threshold — industry convention for pricing data refresh`
- **Missing token data:** If the Claude API response omits `usage` fields, log a warning (per P45: warn on missing numeric data) and record `estimatedCostUsd: null` in the run record rather than silently defaulting to 0.

**Visible Progress (`server/lib/progress.ts`):**
- ProgressReporter: dynamic stage list (built at runtime based on tier/config)
- Stderr: `forge_plan: [2/4] Running critic round 1...` (progress bar with current activity)
- `fail()` method for partial progress on error
- Output: `{stages: [{name, durationMs, status}]}` — status is completed/failed/skipped

**Agent Decision Audit Trail (`server/lib/audit.ts`):**
- AuditLog: `{timestamp, stage, agentRole, decision, reasoning}`
- Persisted to `.forge/audit/{tool}-{timestamp}.jsonl` (Windows-safe, no colons)
- Failure policy: warn and continue, never crash the tool
- 1000-file warning threshold: at ~2KB per audit file, 1000 files is ~2MB — negligible on disk but signals high usage. When the threshold is hit, the warning recommends the user archive or delete old audit files (e.g., `rm .forge/audit/*-2025-*`). No auto-rotation — the user decides.

**RunContext (`server/lib/run-context.ts`):**
- Bundles CostTracker + ProgressReporter + AuditLog
- `callClaude` stays pure — `trackedCallClaude` wrapper handles observability
- Each tool handler creates RunContext at top
- **Migration:** All existing `callClaude` call sites in `plan.ts` (4 calls: planner initial, planner validation-retry, critic, corrector) migrate to `trackedCallClaude` during Step 2. New call sites in Steps 4-5 use `trackedCallClaude` from the start.

### D7: Run Records for Self-Improvement (All Primitives)

Like skills (runs/data.json, run.log), each forge primitive tracks every invocation. This enables learning across runs — detecting patterns, measuring improvement, and informing future decisions.

**Per-primitive run record:**
```json
{
  "timestamp": "ISO-8601",
  "tool": "forge_plan | forge_evaluate | forge_generate | forge_coordinate",
  "documentTier": "master | phase | update | null",
  "mode": "feature | bugfix | full-project",
  "tier": "quick | standard | thorough",
  "metrics": {
    "inputTokens": 0, "outputTokens": 0, "estimatedCostUsd": 0,
    "critiqueRounds": 2, "findingsTotal": 8,
    "findingsApplied": 7, "findingsRejected": 1,
    "validationRetries": 0, "durationMs": 45000
  },
  "outcome": "success | validation-failure | api-error | timeout"
}
```

**Storage:** `.forge/runs/{tool}-{timestamp}-{suffix}.json` (one JSON file per invocation, Windows-safe naming — no colons in filenames). The 4-character random suffix handles same-millisecond collisions on concurrent invocations.

**Self-improvement loop:** Over time, run records reveal:
- Which modes/tiers produce the most critique findings (calibration signal)
- Average cost per tier (budget estimation)
- Validation retry rates (prompt quality signal)
- Trends in finding severity (are plans getting better?)

**Schema versions are per-type and independent.** MasterPlan starts at 1.0.0 and ExecutionPlan is at 3.0.0 because ExecutionPlan has had prior versions. Both types version independently and can coexist in the same run.

### D10: forge_evaluate Sets Guardrails for ALL Primitives

The evaluation primitive is not just a story-level AC runner — it is the quality gatekeeper for the entire forge system. Each primitive's output passes through forge_evaluate before being trusted.

**What forge_evaluate guards:**

| Primitive | What it produces | What evaluate checks | Guardrail type |
|-----------|-----------------|---------------------|----------------|
| forge_plan (master) | MasterPlan JSON | Vision coverage, phase sequencing, input/output chains | Coherence eval (LLM-judged) |
| forge_plan (phase) | ExecutionPlan JSON | Phase alignment, AC quality (functional not implementation-coupled), coverage | Coherence eval + Tier 1 regex |
| forge_plan (update) | Updated plan JSON | Consistency of updates, divergence capture completeness, no silent degradation | Coherence eval |
| forge_generate | Code changes | AC pass/fail (mechanical), unplanned capabilities (reverse divergence) | Story eval + Divergence eval |
| forge_coordinate | Orchestration decisions | Budget compliance, story ordering correctness, blocked-phase handling | Run record metrics (deferred — coordinate is a stub until it is implemented) |

**Guardrail enforcement tiers (per constitution.md P13):**
- **Tier 1 (mechanical, 100% compliance):** Schema validation, AC regex scan for implementation coupling, binary AC pass/fail execution
- **Tier 2 (named rules + consequences):** Coherence eval dimensions (enumerated check list, structured JSON output), divergence thresholds (>50% method divergence = flag). Note: budget enforcement is a coordinate-level policy (D6), not an evaluate guardrail.
- **Tier 3 (measured + calibrated):** Run record trending (calibration protocol from D4), cross-run pattern detection
- **Tier 4 (behavioral prose, <50% compliance):** Critic prompt dimensions (backup for what Tier 1-2 don't catch)

**What evaluate does NOT guard (human responsibility):**
- PRD quality (human approves the PRD directly via /prd interview)
- Intentional scope changes (human decides to pivot)
- Tool selection (human chooses which tools forge_generate has access to)

**Guardrail configuration:** Each primitive declares what guardrails it needs when registering. The evaluate primitive reads the constitution.md enforcement tiers and applies the appropriate level. This ensures new primitives automatically get guardrails without manual wiring.

### Error Handling for New LLM Calls

All new LLM call patterns follow the existing graceful degradation established in `plan.ts`:
