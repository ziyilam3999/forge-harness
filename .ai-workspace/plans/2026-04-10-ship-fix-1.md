# Ship-fix micro-plan: PR #126 stateless review iteration 1

## Context

PR #126 (`docs(coordinate): add PH04-US-05 spec-vs-types vocabulary check`) failed stateless review with 2 MAJOR bugs. Both are factual/structural — neither is interpretive.

## ELI5

The new story's parser was told to look for types in only ONE folder (`server/types/`), but the most important type it needs to check (`RunRecord`) lives in a DIFFERENT folder (`server/lib/`). So the check would miss most of the bugs it was supposed to catch. Also, one of the ACs has a broken shell command that reports "tests passed" even when they didn't. Both fixes are in the phase plan JSON file only — no PRD or master plan changes needed.

## Bugs to fix

### B1 (MAJOR) — parseTypeDefinitions source-dir contract is too narrow

**Current state:** The PH04-US-05 description and AC02 say `parseTypeDefinitions(typesDir)` walks `server/types/` and builds a Map containing `EvalReport, RunRecord, Story, Phase`. But:

- `EvalReport` lives at `server/types/eval-report.ts` ✓
- `Story`, `Phase` live at `server/types/execution-plan.ts` and `server/types/master-plan.ts` ✓
- **`RunRecord` lives at `server/lib/run-record.ts` ✗** (not in `server/types/`)

If the parser only walks `server/types/`, every `` `RunRecord.X` `` reference in the PRD would be classified as `unknown-type` instead of `unknown-field`, defeating the entire purpose of the check (RunRecord is the most-referenced type in the PRD — `RunRecord.storyId`, `RunRecord.evalVerdict`, `RunRecord.metrics`, `RunRecord.metrics.estimatedCostUsd`).

**Fix:** Change the function contract from `parseTypeDefinitions(typesDir)` to `parseTypeDefinitions(sourceDirs: string[])` (an explicit list of source roots). Story description, AC02, and AC07 all need to mention `["server/types", "server/lib"]` (or similar broader set) as the input. The parser walks each root, finds every exported `interface` / `type` declaration via regex, and builds the unified type map. Excludes `*.test.ts` files.

### B2 (MAJOR) — AC12 shell command is broken

**Current state:** `npx vitest run 2>&1 | grep -q 'passed' && ! grep -q 'failed'`

The second `grep -q 'failed'` has no stdin (the `&&` ends the previous pipeline), so it reads the terminal or returns trivially-true on empty stdin. The negation makes the AC trivially-pass regardless of test outcome.

**Fix:** Capture vitest output once, then test both conditions against the captured string. Pattern:

```
OUT=$(npx vitest run 2>&1); echo "$OUT" | grep -q 'passed' && ! echo "$OUT" | grep -q ' failed'
```

The space prefix on `' failed'` matches vitest's summary line `X failed` without false-matching test names containing the substring.

**Note:** The pre-existing PH04-US-03-AC08 has the same broken pattern. NOT fixing it in this PR — out of scope, will file a follow-up issue.
