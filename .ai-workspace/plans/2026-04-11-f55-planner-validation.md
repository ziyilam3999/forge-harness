# Plan: F55 Planner Validation — Regenerate Phase Plans via forge_plan

## ELI5

We found that the phase plan ACs (acceptance criteria) were hand-written by a Claude Code session, not generated by forge_plan. They're full of bugs (F55 pattern — grep on TTY-dependent output). Instead of hand-fixing 55+ broken ACs, we'll make forge_plan regenerate them. If the planner produces clean ACs, the v0.17.1 AC Command Contract works. If it still produces broken ones, we fix the planner. Either way we learn something — hand-fixing teaches us nothing.

## Context

- **F55:** AC `grep -q 'passed'` on vitest output fails in forge_evaluate's non-TTY subprocess
- **v0.17.1 (PR #134):** Added AC Command Contract to `server/lib/prompts/planner.ts` — tells the LLM to use exit-code checks, not output parsing
- **Problem:** PH-03 and PH-04 phase plans were hand-authored in S2 (commit `b57aa2b`, 2026-04-09) before the contract existed. ~58 ACs use the fragile `grep -q 'passed'` pattern (29 in each file)
- **Root cause:** Two AC authorship paths exist — forge_plan (now fixed) and Claude Code sessions (unfixed). Rather than adding templates for the manual path, we should close it by using forge_plan for all phase plan generation
- **PH-02 recurrence:** PH02-US-01 AC01 failed due to multi-line function signature grep — same F55 family

## Approach

Regenerate PH-03 and PH-04 phase plans using `forge_plan`, then evaluate them. This validates the planner end-to-end and replaces hand-written ACs with machine-generated ones.

## Steps

### Step 1: Regenerate PH-03 via forge_plan

Call `forge_plan` MCP tool with:
- `documentTier`: `"phase"` (required — selects phase plan generation mode)
- `visionDoc`: contents of `docs/forge-coordinate-prd.md` (read file, pass as string)
- `masterPlan`: contents of `.ai-workspace/plans/forge-coordinate-master-plan.json` (read file, pass as JSON string)
- `phaseId`: `"PH-03"`
- `projectPath`: `.` (current project root, enables persistence and codebase scanning)

The planner should generate a phase plan JSON with AC commands that follow the AC Command Contract (exit-code based, no TTY-dependent grep).

**Output:** Returned as MCP response content (JSON string). Save to a temp file for comparison. A RunRecord is also written to `.forge/runs/`.

### Step 2: Compare old vs new PH-03 ACs

Diff the AC `command` fields between:
- Old: `.ai-workspace/plans/forge-coordinate-phase-PH-03.json` (hand-written, F55-vulnerable)
- New: regenerated output from Step 1

**Pass criteria:**
- Zero instances of `2>&1 | grep -q 'passed'` in the new plan
- Zero instances of `grep -qE 'Tests.*passed'` or similar count-based output parsing
- All ACs use exit-code based checks (vitest exit 0 = pass) or captured-output patterns from the AC Command Contract

**Fail criteria:**
- Any F55 pattern in the new output → planner prompt needs strengthening (go to Step 5)

### Step 3: Regenerate PH-04 via forge_plan

Same as Step 1 but with `phaseId: "PH-04"` and `documentTier: "phase"`.

### Step 4: Compare old vs new PH-04 ACs

Same comparison as Step 2.

### Step 5: Decision branch

**If Steps 2+4 pass** (planner produces clean ACs): proceed to Step 6.

**If Steps 2 or 4 fail** (planner still produces F55 ACs):
- Diagnose: read the AC Command Contract in `server/lib/prompts/planner.ts` (line 272) and identify why it's not preventing the pattern
- Fix the planner prompt — strengthen the contract
- Re-run Steps 1-4 to verify the fix
- Then proceed to Step 6

### Step 6: Coherence verification

Replace `.ai-workspace/plans/forge-coordinate-phase-PH-03.json` and `PH-04.json` with the regenerated versions, then run `forge_evaluate` in coherence mode:
- `prdPath`: `docs/forge-coordinate-prd.md`
- `masterPlanPath`: `.ai-workspace/plans/forge-coordinate-master-plan.json`
- `phasePlanPaths`: all 4 phase plan JSONs (PH-01 + PH-02 unchanged, PH-03 + PH-04 regenerated)
- `projectPath`: `.`

**Pass criteria:** Coherence PASS with 0 Critical, 0 Major findings.

### Step 7: Ship

- Commit regenerated phase plans on branch, PR via `/ship`
- Update F55 anti-pattern entry in `hive-mind-persist/knowledge-base/02-anti-patterns.md` with resolution evidence

## Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| Planner generates different story structure than hand-written plans | MEDIUM | Compare story IDs and AC count before replacing. The stories and their intent should match — only the AC commands should differ |
| Planner AC Contract still insufficient | MEDIUM | This is the point of the exercise — if it fails, we fix the prompt and re-run. Failure is signal, not waste |
| Regenerated ACs are correct but test different things than originals | LOW | Review AC intent, not just format. The story descriptions and the PRD are the source of truth for what to test |
| forge_plan MCP server stale after rebuild (F54) | LOW | Restart session if MCP tools fail. Already documented in memory |

## Results

### F55 Pattern Elimination: PASS

| Check | Old PH-03 | New PH-03 | Old PH-04 | New PH-04 |
|-------|-----------|-----------|-----------|-----------|
| `grep -q 'passed'` | 29 | **0** | 29 | **0** |
| `grep -qE.*passed` | 0 | 0 | 0 | 0 |
| `2>&1 \| grep` | 29 | **0** | 27 | **2** (stderr checks, not TTY) |

**Conclusion:** The AC Command Contract in `planner.ts:272-283` (v0.17.1, PR #134) successfully prevents the F55 pattern. The planner generates exit-code-based ACs using `node -e` with direct module imports instead of piping test runner output to grep.

The 2 remaining `2>&1 | grep` in PH-04 are stderr warning checks (checking P45 warning emission), not TTY-dependent test output parsing.

### Story Structure: Preserved

- PH-03: 5 stories (old) → 5 stories (new)
- PH-04: 6 stories (old) → 6 stories (new)

### Coherence Eval: 0C/5M/3m

Gaps are all structural/metadata, NOT F55-related:
- MAJOR: Story ID naming (`US-xx` vs `PH03-US-xx` convention)
- MAJOR: Missing `masterPlanPath`/`prdPath` metadata fields
- MAJOR: AC style uses `dist/` requires instead of `npx vitest run`
- MAJOR: Empty dependency arrays where ordering is needed
- MAJOR: Missing readAuditEntries graceful-degradation ACs

These structural gaps exist because the planner's system prompt doesn't enforce project-specific conventions. The hand-written plans had these because the human author followed the conventions.

### Decision

**Do NOT replace phase plans with regenerated versions.** The originals have correct structure, metadata, and dependency chains. The regenerated plans validate the planner's AC quality but lack project-specific conventions. For PH-03/PH-04 implementation, use the original hand-written plans with manually fixed ACs (replace `grep -q 'passed'` patterns with exit-code-based equivalents per the AC Command Contract).

Regenerated plans saved for reference at `.ai-workspace/plans/f55-validation/`.

### MCP Auth Finding

The forge MCP server process (spawned by Claude Code at session start) did not inherit `ANTHROPIC_API_KEY`, falling through to OAuth which returned 401. Direct node invocation worked. Workaround: restart session after setting env var, or invoke planner directly via node script.
