## ELI5

Imagine the "evaluate" tool is a doctor that checks whether each test in your test plan actually works. Today it only grades PASS/FAIL. We're adding a second mode — "smoke-test" — that doesn't grade at all. It just *asks each test*: "Can you even start? Do you finish on time? Do you produce any output?" And writes a little report card: `ok`, `slow`, `hung`, `empty-evidence`, or `skipped-suspect` (the last one means "a lint check already flagged this test as dangerous, so we didn't even run it"). This catches broken tests at authoring time, before they ever get near real code. It's like a fire drill: we're not checking if the building survives a fire, we're checking if the alarm even rings.

**Why now:** two of PH-01's ACs are known-bad (F55/F56 class — subprocess patterns that produce empty evidence or hang). Layer A catches them at lint time (shape check). Layer B catches them at smoke time (runtime characterization). Defense in depth: if an AC slips past the lint regex, the smoke-test still flags it.

## Design decisions (locked)

### D1. Raw `smokeExecute` vs reusing `executeCommand`

**Locked: add new `smokeExecute`. Do NOT reuse `executeCommand`.**

Reason: `executeCommand` returns `CriterionResult` with `status: PASS|FAIL|INCONCLUSIVE` and truncated evidence. Smoke-test needs raw wall-clock, raw byte counts, and explicit `hungOnTimeout` flag. Reusing `executeCommand` and post-processing would require the smoke-runner to guess which FAILs were "real fails" vs timeouts — info that the existing function discards. Cleaner seam: a separate low-level function that returns the raw data, and the smoke-runner does classification.

Shared code: both functions call the same `resolveWindowsBashPath` helper and use the same `exec` + `shell` pattern. Code duplication is ~30 lines of exec wiring — acceptable to avoid conditional return shapes.

### D2. Timeout precision, clamp rules, and Windows warmup

**Locked: use `exec`'s native `timeout` option for hard kill; record elapsed via `Date.now()` bracketing.**

Why: `exec`'s timeout fires a SIGTERM → SIGKILL escalation. We treat `error.killed === true` (via the child_process callback) as the `hungOnTimeout: true` signal. No drift concerns — the kill is enforced at the OS level, not via setTimeout coercion.

**`smokeTimeoutMs` clamp rules (promoted from tests 13/14 per F7 fix):**
- `undefined` or `null` → default 30000ms
- `NaN` or `!Number.isFinite(v)` → default 30000ms
- `v < 0` → default 30000ms (negative treated as "use default", not zero — prevents pathological instant-fail)
- `v === 0` → default 30000ms (same reason)
- `v > 180000` → 180000ms (hard cap)
- otherwise → v

Clamp happens in the smoke-runner before calling `smokeExecute`. `smokeExecute` trusts its input.

**Windows cold-start subtraction:** applied in the smoke-runner AFTER `smokeExecute` returns, only to the first AC's `elapsedMs` output field. Floor at zero (if `rawElapsed < 800`, emit 0, not a negative). `smokeExecute` itself always returns raw wall-clock — no `rawElapsedMs` alias field. Tests stub `smokeExecute` with an `elapsedMs` value and assert the smoke-runner's post-subtraction output.

### D3. `skipped-suspect` classification routing

**Locked: ac-lint check happens in the smoke-runner loop, NOT in handleSmokeTest router.**

Why: the plan spec says "Every AC in the plan MUST have a corresponding entry" — the smoke-runner is the single producer of entries. If handleSmokeTest routed some ACs to a separate "skipped" path, we'd have two sites that can forget to emit an entry. One loop, one emission site, one invariant.

### D4. `smoke-gate.yml` report-only in this PR

**Locked: report-only (no merge-block) in B1's landing PR.**

Why: mirrors the A1c → C1 split pattern the plan explicitly endorses. B1 ships the detection infrastructure; the C1-analog binding PR ships the merge gate. Bootstrap self-exemption is still exercised in this PR (the landing commit proves the `handleSmokeTest`-absent-on-master arm of the guard). Binding-mode testing belongs in the follow-up PR where we can use a synthetic "plan edit without smoke report" as the failing case.

### D5. `lintExempt` wins (plan line 177 precedence rule)

**Locked: exempt ACs execute normally in smoke-test, regardless of pattern match.**

Matches the A1b+A3 precedence decision already in the plan at line 177. Test case pins this explicitly (see Test 9 below).

## Test Cases & AC (binary exit criteria)

Every test assertion is strictly binary. No "reasonable" / "about" / "should be".

1. **ok verdict — clean fast command:** `flaky: false, command: "echo hi"` → verdict `ok`, `exited=0`, `elapsedMs < 24000` (80% of 30s default), `evidenceBytes > 0`, `timeoutRisk=false`.
2. **slow verdict without override:** stub `smokeExecute` to return `{exitCode: 0, elapsedMs: 25000, stdoutBytes: 10, stderrBytes: 0, hungOnTimeout: false}` with default timeout=30000 → report entry `{verdict: "slow", exited: 0, timeoutRisk: true}`.
3. **slow verdict with explicit override:** same stub, `ac.smokeTimeoutMs=30000` → verdict `slow`, `timeoutRisk=false` (override consent suppresses the risk flag).
4. **hung verdict:** stub returns `{exitCode: null, hungOnTimeout: true, ...}` → report entry `{verdict: "hung", exited: null, timeoutRisk: false}` (hung is its own verdict, not a modifier).
5. **empty-evidence verdict:** stub returns `{exitCode: 1, stdoutBytes: 0, stderrBytes: 0, hungOnTimeout: false, elapsedMs: 100}` → report entry `{verdict: "empty-evidence", exited: 1}`.
6. **skipped-suspect verdict — lint flagged:** AC command matches F55 deny pattern (e.g. `"node -e 'console.log(1)' > /dev/null"`) → verdict `skipped-suspect`, `reason` contains the rule id, `exited=null`, `elapsedMs=null`, `evidenceBytes=null`, `timeoutRisk=false`, `smokeExecute` NEVER called (spy assertion).
7. **skipped-suspect comes BEFORE any execution attempt:** mock `smokeExecute` to throw; lint-flagged AC still produces clean `skipped-suspect` entry (proves the structural bypass, not a caught exception).
8. **every AC gets an entry (completeness invariant):** plan with 3 stories × 2 ACs each = 6 ACs, mixed (1 ok, 1 slow, 1 hung, 1 empty-evidence, 1 skipped-suspect, 1 ok) → `entries.length === 6` AND every `acId` from the plan appears exactly once.
9. **lintExempt wins:** AC has `command: "<F55 pattern>"` AND `lintExempt: [{ruleId: "F55", rationale: "test"}]` → `smokeExecute` IS called (not skipped-suspect), verdict derived from actual run.
10. **windows cold-start subtraction:** platform-independent test via an explicit `platformOverride: "win32"` option on `smokeTestPlan` (test-only; production callers never pass it). Stub `smokeExecute` to return `{elapsedMs: 1000, exitCode: 0, ...}` for BOTH first and second AC → first AC's report entry has `elapsedMs=200`, second has `elapsedMs=1000`. Floor test: stub `elapsedMs: 500` for first AC → report entry `elapsedMs=0` (not -300). Runs on all platforms via the override — no `it.runIf` skip, no platform-dependent test count.
11. **sidecar write:** `handleSmokeTest` with `planPath: join(os.tmpdir(), "test-plan.json")` writes the matching `.smoke.json` file next to it. Serialization is `JSON.stringify(report, null, 2)` with entries sorted by `acId` for determinism (no full canonical-key-sort helper — that's overkill for a flat report shape). Assertion: round-trip `JSON.parse(readFile())` equals in-band return. Uses `os.tmpdir()` for Windows/POSIX portability — NO hardcoded `/tmp`.
12. **sidecar skip for inline plan:** `handleSmokeTest` with `planJson: "..."` (no planPath) returns the report in-band and does NOT attempt any FS write.
13. **timeout cap:** AC with `smokeTimeoutMs: 300000` (5 min) → spy on `smokeExecute` asserts it was called with `timeoutMs: 180000` (clamped to D2 hard cap). No warning field assertion (SmokeReport has no metadata slot).
14. **timeout floor/sanitize (D2 clamp rules):** four sub-cases in a single test — `smokeTimeoutMs: -5` → 30000; `NaN` → 30000; `0` → 30000; `undefined` → 30000. Each sub-case spies the `smokeExecute` timeoutMs arg.
